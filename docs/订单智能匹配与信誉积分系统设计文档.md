## 一、订单智能匹配（推荐）算法

### 1.1 功能概述

当跑腿员在"接单大厅"中点击**「智能推荐」**时，系统根据跑腿员当前的**地理位置**、订单的**截止时间紧迫度**以及跑腿员自身的**信誉分**，综合计算每个待接单订单与该跑腿员的匹配度，按匹配度从高到低排序，帮助跑腿员快速找到最合适的订单。

### 1.2 触发场景与调用链路

```
用户操作（前端）                    后端处理链路
─────────────                   ──────────────
接单大厅 → 点击「智能推荐」
    ↓ 携带浏览器定位的 lat/lng
    ↓
GET /campus/order/recommend      → ErrandOrderController.recommend()
    ?lat=30.123&lng=120.456         ↓
                                 → ErrandOrderServiceImpl.selectRecommendedOrders()
                                    ├─ ① 校验跑腿员身份（StudentProfileMapper）
                                    ├─ ② 查询所有待接单订单（ErrandOrderMapper）
                                    ├─ ③ 获取跑腿员信誉分
                                    └─ ④ 调用 OrderMatchAlgorithm.calculate()
                                          ├─ 过滤无效订单
                                          ├─ 逐单计算三因子得分
                                          ├─ 加权求和 → 综合匹配分
                                          └─ 按匹配分降序排序 → 返回结果
```

### 1.3 前置校验

在进入算法计算之前，系统执行以下校验：

| 校验项 | 校验逻辑 | 不通过处理 |
|--------|----------|-----------|
| 跑腿员身份 | 查询 `campus_student_profile` 表，`is_runner = '1'` | 抛出异常：`"您没有接单权限，请先完成学生认证"` |
| 待接单订单 | 查询 `order_status = '0'` 的订单列表 | 返回空列表 |

进入算法后，还会对每条订单做以下过滤：

| 过滤条件 | 说明 |
|----------|------|
| `publisherId == runnerId` | 排除跑腿员自己发布的订单 |
| `deadlineTime < 当前时间` | 排除已过截止时间的订单 |

### 1.4 算法模型：三因子加权评分

系统采用 **三因子线性加权模型** 对每个待接单订单进行评分：

```
综合匹配分 = ( 距离因子得分 × 40% + 时间因子得分 × 30% + 信誉因子得分 × 30% ) × 100
```

| 因子 | 权重 | 含义 |
|------|------|------|
| **距离因子** | **40%** | 跑腿员与取货点的物理距离，越近分越高 |
| **时间因子** | **30%** | 订单截止时间的紧迫程度，越紧迫分越高 |
| **信誉因子** | **30%** | 跑腿员的信誉积分，信誉越高分越高 |

> **设计思想**：距离是校园场景中影响接单效率的最关键因素（权重最高 40%），时间紧迫度确保紧急订单优先被匹配，信誉分鼓励高信誉跑腿员获得更好的推荐。

### 1.5 各因子计算细节

#### 1.5.1 距离因子（Distance Score）

**目标**：跑腿员距离取货点越近，得分越高。

**计算步骤**：

1. 使用 **Haversine（半正矢）公式** 计算跑腿员坐标 `(runnerLat, runnerLng)` 与订单取货点坐标 `(pickupLat, pickupLng)` 之间的球面距离（单位：米）。

2. 线性归一化到 `[0, 1]` 区间：

$$
\text{distanceScore} = \max\left(0,\; 1 - \frac{\text{distance(m)}}{\text{MAX\_DISTANCE}}\right)
$$

**Haversine 公式**：

$$
\begin{aligned}
a &= \sin^2\!\left(\frac{\Delta\text{lat}}{2}\right) + \cos(\text{lat}_1) \cdot \cos(\text{lat}_2) \cdot \sin^2\!\left(\frac{\Delta\text{lng}}{2}\right) \\[6pt]
c &= 2 \cdot \text{atan2}\!\left(\sqrt{a},\; \sqrt{1-a}\right) \\[6pt]
d &= R \cdot c
\end{aligned}
$$

其中 $R = 6{,}371{,}000\text{m}$（地球平均半径）。

**归一化基准**：`MAX_DISTANCE = 5000m`（5公里，覆盖校园及周边区域）

| 实际距离 | 距离得分 | 说明 |
|----------|---------|------|
| 0m | 1.0 | 距离为零，满分 |
| 500m | 0.9 | 近距离 |
| 1000m | 0.8 | 步行可达 |
| 2500m | 0.5 | 中等距离 |
| 5000m | 0.0 | 达到基准上限 |
| >5000m | 0.0 | 超出范围，得分为零 |

**缺失坐标处理**：若跑腿员坐标或订单取货点坐标为空，距离得分直接为 `0.0`。

#### 1.5.2 时间因子（Time Score）

**目标**：订单截止时间越紧迫，得分越高（优先推荐即将到期的订单）。

**计算公式**：

$$
\text{remainingMinutes} = \frac{\text{deadlineTime} - \text{now}}{60{,}000}
$$

$$
\text{timeScore} = \max\left(0,\; 1 - \frac{\text{remainingMinutes}}{\text{MAX\_TIME\_WINDOW}}\right)
$$

**归一化基准**：`MAX_TIME_WINDOW = 1440 分钟`（24小时）

| 剩余时间 | 时间得分 | 说明 |
|----------|---------|------|
| 30分钟 | 0.979 | 非常紧急 |
| 1小时 | 0.958 | 紧急 |
| 6小时 | 0.75 | 较紧迫 |
| 12小时 | 0.5 | 一般 |
| 24小时 | 0.0 | 截止时间基准上限 |
| >24小时 | 0.0 | 不紧急，得分为零 |

**特殊情况**：
- 截止时间为空：时间得分为 `0.0`
- 已过截止时间：该订单在前置过滤中已被排除

#### 1.5.3 信誉因子（Credit Score）

**目标**：信誉分越高的跑腿员，综合匹配分越高，获得更优质的推荐排序。

**计算公式**：

$$
\text{creditScore} = \min\!\left(1.0,\; \max\!\left(0.0,\; \frac{\text{rawCreditScore}}{\text{CREDIT\_BASE}}\right)\right)
$$

**归一化基准**：`CREDIT_BASE = 100 分`

| 信誉分 | 信誉得分 | 说明 |
|--------|---------|------|
| 100分 | 1.0 | 满分 |
| 80分 | 0.8 | 良好 |
| 60分 | 0.6 | 一般 |
| 30分 | 0.3 | 较差 |
| 0分 | 0.0 | 最低 |
| >100分 | 1.0 | 上限截断为1.0 |

> **注意**：信誉因子对同一跑腿员来说是固定值（所有订单的该因子得分相同），它影响的是该跑腿员看到的所有订单的综合匹配分的"基线高度"。

### 1.6 综合匹配分计算

将三个因子得分按权重加权求和，乘以 100 映射到 `[0, 100]` 分区间：

$$
\text{matchScore} = \left(\text{distanceScore} \times 0.4 + \text{timeScore} \times 0.3 + \text{creditScore} \times 0.3\right) \times 100
$$

最终结果保留 **2位小数**。

### 1.7 排序与输出

- 按 `matchScore` **降序排列**
- 返回 `List<OrderRecommendationDTO>`，每条记录包含完整的订单信息 + 匹配度相关字段

### 1.8 数据结构

#### OrderRecommendationDTO（推荐结果传输对象）

| 字段 | 类型 | 说明 |
|------|------|------|
| *继承 ErrandOrder 的全部字段* | — | 订单完整信息 |
| `matchScore` | Double | 综合匹配分 (0~100)，分数越高越推荐 |
| `distanceScore` | Double | 距离因子得分 (0~1) |
| `timeScore` | Double | 时间因子得分 (0~1) |
| `creditScore` | Double | 信誉因子得分 (0~1) |
| `distanceToPickup` | Double | 跑腿员与取货点的距离（米） |

### 1.9 算法示例

**场景**：跑腿员小张（信誉分 85）在教学楼 A（lat=30.5, lng=120.3）查看智能推荐。

| 订单 | 取货点距离 | 截止时间剩余 | 距离得分 | 时间得分 | 信誉得分 | **综合分** |
|------|-----------|-------------|---------|---------|---------|----------|
| 订单A | 800m | 2小时 | 0.84 | 0.917 | 0.85 | **0.84×40 + 0.917×30 + 0.85×30 = 86.61** |
| 订单B | 200m | 18小时 | 0.96 | 0.25 | 0.85 | **0.96×40 + 0.25×30 + 0.85×30 = 71.40** |
| 订单C | 3000m | 45分钟 | 0.40 | 0.969 | 0.85 | **0.40×40 + 0.969×30 + 0.85×30 = 70.57** |

**推荐排序结果**：订单A (86.61) > 订单B (71.40) > 订单C (70.57)

> 分析：订单 A 兼具近距离和较紧迫的截止时间，综合表现最佳。订单 B 虽然最近，但时间不紧迫拉低了分数。订单 C 虽然最紧迫，但距离太远。

### 1.10 参数配置一览

|--------|-----|------|--------|
| `WEIGHT_DISTANCE` | 0.4 | 距离权重 40% | `OrderMatchAlgorithm` |
| `WEIGHT_TIME` | 0.3 | 时间权重 30% | `OrderMatchAlgorithm` |

## 二、订单智能定价算法

### 2.1 功能概述

用户发布跑腿订单时，系统基于**高德地图 API 规划的真实步行路径距离**，结合**物品重量类型**和**加急需求**，由后台算法自动生成建议的跑腿费用。

这一机制解决了传统模式下用户盲目定价带来的两大问题：
- **发布者定价过低**：跑腿员无人愿接，订单长期滞留
- **发布者定价过高**：造成不必要的经济负担

系统生成的建议价格仅作为参考，用户仍可在此基础上手动调整。

### 2.2 触发场景与调用链路

```
用户操作（前端）                              后端/API 处理链路
─────────────                              ──────────────
发布订单 → 选定取货点坐标
         → 选定送达点坐标
              ↓
         前端自动调用高德步行路径规划
         AMap.Walking.search(origin, dest)
              ↓
         获得真实路径距离 routeDistance(米)
              ↓
         GET /campus/order/calcPrice        → ErrandOrderController.calcPrice()
             ?distanceMeters=1850               ↓
             &weightCategory=MEDIUM          → PricingAlgorithm.calculate()
             &isUrgent=1                        ├─ ① 阶梯距离费
                                                ├─ ② 重量附加费
                                                ├─ ③ 加急倍率
                                                └─ ④ 返回建议价格
              ↓
         前端自动填入「建议价格」字段
         用户可进行手动微调
```

**自动重新计算**：当用户修改**重量类型**或**是否加急**选项时，前端通过 `watch` 监听器自动重新调用定价接口，无需用户手动触发。

### 2.3 定价模型：阶梯距离 + 重量附加 + 加急倍率

系统采用**复合定价模型**，总价由三个维度叠加构成：

$$
\text{suggestedPrice} = \left(\text{baseFee} + \text{distanceFee} + \text{weightSurcharge}\right) \times \text{urgentMultiplier}
$$

| 费用项 | 说明 |
|--------|------|
| **baseFee（基础服务费）** | 起步价，包含首 1 公里的服务 |
| **distanceFee（距离费）** | 超出 1 公里部分按阶梯费率计算 |
| **weightSurcharge（重量附加费）** | 根据物品重量类型额外收费 |
| **urgentMultiplier（加急倍率）** | 加急订单在小计基础上乘以倍率 |

> **设计思想**：基础费保障跑腿员的最低劳动报酬；阶梯距离费体现"距离越远成本越高"的合理性；重量附加费补偿搬运大件物品的体力消耗；加急倍率则是对时效服务的溢价补偿。

### 2.4 各费用项计算细节

#### 2.4.1 基础服务费（Base Fee）

| 项目 | 值 |
|------|-----|
| 基础费 | **¥3.00** |
| 包含范围 | 首 1 公里（含）以内的跑腿服务 |

#### 2.4.2 距离费（Distance Fee）— 阶梯计算

距离费仅对**超出首 1 公里**的部分收取，采用两级阶梯费率：

| 距离区间 | 单价 | 说明 |
|----------|------|------|
| 0 ~ 1 km | ¥0（含在基础费中） | 起步范围 |
| 1 ~ 3 km | **¥1.50/km** | 常规校园范围 |
| > 3 km | **¥2.00/km** | 超出校园核心区 |

**计算公式**（设超出首公里的距离为 $d_{over}$ km）：

$$
\text{distanceFee} = \begin{cases}
0 & d_{over} \leq 0 \\[4pt]
1.5 \times d_{over} & 0 < d_{over} \leq 2 \\[4pt]
1.5 \times 2 + 2.0 \times (d_{over} - 2) & d_{over} > 2
\end{cases}
$$

#### 2.4.3 重量附加费（Weight Surcharge）

| 重量类型 | 编码 | 附加费 | 典型物品 |
|----------|------|--------|---------|
| 小件 | `SMALL` | **¥0** | 文件、钥匙、小包裹 |
| 中件 | `MEDIUM` | **¥2.00** | 外卖、书本、小箱子 |
| 大件 | `LARGE` | **¥5.00** | 大箱快递、桶装水 |

#### 2.4.4 加急倍率（Urgent Multiplier）

| 类型 | 倍率 | 说明 |
|------|------|------|
| 普通 (`isUrgent = "0"`) | **×1.00** | 不加价 |
| 加急 (`isUrgent = "1"`) | **×1.30** | 总价上浮 30% |

#### 2.4.5 最低价格保护

无论计算结果如何，最终价格不低于 **¥3.00**（即基础费）。

### 2.5 定价示例

| 场景 | 路径距离 | 重量 | 加急 | 计算过程 | **建议价格** |
|------|---------|------|------|---------|------------|
| 短距离小件 | 500m | 小件 | 否 | (3 + 0 + 0) × 1.0 | **¥3.00** |
| 校园内中件 | 1.8km | 中件 | 否 | (3 + 0.8×1.5 + 2) × 1.0 | **¥6.20** |
| 中距离加急 | 2.5km | 小件 | 是 | (3 + 1.5×1.5 + 0) × 1.3 | **¥6.83** |
| 较远大件加急 | 4.2km | 大件 | 是 | (3 + 2×1.5 + 1.2×2.0 + 5) × 1.3 | **¥17.42** |
| 极短距离 | 200m | 大件 | 否 | (3 + 0 + 5) × 1.0 | **¥8.00** |

**详细推演（场景4：较远大件加急）**：

```
路径距离 = 4.2km
超出首公里 = 4.2 - 1 = 3.2km
  其中 1~3km 部分 = 2km → 2 × 1.5 = ¥3.00
  其中 >3km 部分 = 1.2km → 1.2 × 2.0 = ¥2.40
距离费 = 3.00 + 2.40 = ¥5.40
小计 = 3.00(基础) + 5.40(距离) + 5.00(大件) = ¥13.40
加急 = 13.40 × 1.30 = ¥17.42
```

### 2.6 路径距离获取：高德步行路径规划

系统使用**高德地图 JS API 2.0** 的 `AMap.Walking`（步行路径规划）服务获取真实的路径距离，而非简单的直线距离。

**为什么用步行规划而非直线距离？**

| 方式 | 取货点→送达点 500m 直线 | 实际情况 |
|------|----------------------|---------|
| 直线距离（Haversine） | 500m | 未考虑建筑物、围墙、校园道路弯绕 |
| **步行路径规划（Walking）** | **约 750m** | 沿实际道路计算，更贴近真实步行距离 |

**调用方式（前端）**：

```javascript
AMap.plugin('AMap.Walking', function() {
    var walking = new AMap.Walking();
    walking.search(
        [pickupLng, pickupLat],        // 起点：取货点
        [deliveryLng, deliveryLat],     // 终点：送达点
        function(status, result) {
            if (status === 'complete') {
                var meters = result.routes[0].distance;  // 步行路径距离(米)
            }
        }
    );
});
```

### 2.7 前端自动触发机制

定价过程对用户完全透明，无需手动操作：

```
┌────────────────────────────────────────────────────────┐
│                   订单发布表单                           │
├────────────────────────────────────────────────────────┤
│                                                        │
│  ① 选择取货点 ──┐                                      │
│                  ├─→ 两点都选定 ─→ AMap.Walking         │
│  ② 选择送达点 ──┘       ↓          获取路径距离          │
│                         ↓                              │
│                    GET /calcPrice ─→ 自动填入建议价格     │
│                         ↑                              │
│  ③ 修改重量类型 ────────┤   (watch 监听自动重新计算)      │
│  ④ 修改是否加急 ────────┘                               │
│                                                        │
│  ⑤ 建议价格框 ← [自动填入，可手动微调]                    │
│     提示：「算法自动生成，您也可以手动调整」                │
│                                                        │

```

| 修改重量类型（watch） | 仅调用定价接口（距离不变） |
| 修改是否加急（watch） | 仅调用定价接口（距离不变） |

## 三、用户信誉积分系统

### 3.1 功能概述

信誉积分系统是校园跑腿平台的**信任基石**，通过对跑腿员的行为进行量化评价，建立一套**奖惩分明**的积分机制：

- **按时完成订单** → 加分奖励
- **超时完成订单** → 扣分惩罚
- **接单后取消订单** → 重度扣分

信誉分直接影响跑腿员的**接单权限**和**智能推荐排名**。

### 3.2 积分初始化

| 项目 | 说明 |
|------|------|
| 初始分值 | **100 分**（数据库 `campus_student_profile.credit_score` 默认值） |
| 分值范围 | 无硬性上下限（理论上可低于 0 或超过 100） |
| 生效时机 | 学生完成认证、管理员审核通过后即拥有初始信誉分 |

### 3.3 积分变更规则

系统定义了三种信誉变更类型：

| 变更类型 | 编码 | 触发条件 | 分值变动 | 说明 |
|----------|------|---------|---------|------|
| **按时完成** | `COMPLETE` | 跑腿员在截止时间前完成订单 | **+2** | 鼓励准时交付 |
| **超时完成** | `LATE` | 跑腿员在截止时间后完成订单 | **-3** | 惩罚延迟行为 |
| **违规取消** | `VIOLATION` | 跑腿员接单后（状态为"已接单"）订单被取消 | **-5** | 严惩不负责任的弃单行为 |

> **设计原则**：扣分力度 > 加分力度，鼓励跑腿员慎重接单、按时履约。

### 3.4 积分变更触发时机

#### 3.4.1 完成订单时（`completeOrder`）

```
触发入口：ErrandOrderServiceImpl.completeOrder(orderId)

执行流程：
  ① 校验订单状态必须为 "2"（配送中）
  ② 更新订单状态为 "3"（已完成），记录完成时间
  ③ 判断是否超时：
     ├─ deadlineTime != null && 当前时间 > deadlineTime
     │   → 超时：changeType = "LATE"，scoreChange = -3
     └─ 否则
         → 按时：changeType = "COMPLETE"，scoreChange = +2
  ④ 计算新积分：newScore = runner.creditScore + scoreChange
  ⑤ 更新 campus_student_profile 表的 credit_score 字段
  ⑥ 插入 campus_credit_log 记录
```

**超时判定逻辑详解**：

```java
boolean isLate = order.getDeadlineTime() != null && new Date().after(order.getDeadlineTime());
```

- 比较的是**点击"完成"按钮的时刻** vs **订单设定的截止时间**
- 如果订单没有设置截止时间（`deadlineTime == null`），视为**不超时**，正常加 2 分

#### 3.4.2 取消订单时（`cancelOrder`）

```
触发入口：ErrandOrderServiceImpl.cancelOrder(orderId)

执行流程：
  ① 校验订单状态不能为 "3"（已完成）或 "4"（已取消）
  ② 更新订单状态为 "4"（已取消）
  ③ 判断是否需要扣分：
     ├─ 条件：runnerId != null（已有人接单）且 原状态为 "1"（已接单）
     │   → changeType = "VIOLATION"，scoreChange = -5
     │   → 更新跑腿员信誉分，插入日志
     └─ 否则（未被接单的取消）
         → 不产生信誉变更
```

**关键判断**：只有**已被接单（状态="1"）**后的取消才会扣分。待接单（状态="0"）状态下的取消不影响任何人的信誉分。

### 3.5 积分日志记录

每次信誉分变更时，系统自动向 `campus_credit_log` 表插入一条记录：

```java
CreditLog creditLog = new CreditLog();
creditLog.setUserId(runnerId);          // 被影响的跑腿员ID
creditLog.setChangeType(changeType);     // COMPLETE / LATE / VIOLATION
creditLog.setScoreChange(scoreChange);   // +2 / -3 / -5
creditLog.setCurrentScore(newScore);     // 变更后的当前分数
creditLog.setRefOrderId(orderId);        // 关联的订单ID（可追溯）
```

日志特点：
- **只增不改**：日志记录一旦插入不会被修改，确保审计可追溯
- **关联订单**：通过 `refOrderId` 可追溯到具体是哪个订单引发的积分变更
- **快照记录**：`currentScore` 记录的是变更后的即时分数，可重建积分变化曲线

### 3.6 积分与系统功能联动

信誉积分不是孤立存在的，它与系统其他功能深度联动：

```
┌─────────────────────────────────────────────────────┐
│                    信誉积分系统                        │
│                  credit_score                         │
└──────────┬───────────────┬───────────────┬───────────┘
           │               │               │
     ┌─────▼─────┐  ┌─────▼──────┐  ┌────▼──────┐
     │ 接单权限   │  │ 智能推荐    │  │ 前端展示   │
     │ 校验       │  │ 排序因子    │  │ 颜色标记   │
     └───────────┘  └────────────┘  └───────────┘
```

| 联动功能 | 说明 |
|----------|------|
| **接单权限校验** | 接单和查看推荐时，校验 `is_runner = '1'`（需通过学生认证），信誉分本身不直接限制接单，但影响推荐排名 |
| **智能推荐排序** | 信誉分通过归一化后作为 30% 权重参与匹配分计算，信誉越高推荐越靠前 |
| **前端颜色展示** | 信誉分 ≥80 显示绿色（健康），60~79 显示橙色（警告），<60 显示红色（危险） |

## 四、三大系统的联动关系

订单智能定价、智能匹配和信誉积分三大算法并非独立运行，而是形成了一个**闭环联动**的系统：

```
                    ┌──────────────────────┐
                    │    ① 智能定价算法     │
                    │    PricingAlgorithm   │
                    └──────────┬───────────┘
                               │ 自动生成合理价格
                               ▼
                    ┌──────────────────────┐
                    │  发布者发布订单       │
                    │  (价格合理,吸引接单)  │
                    └──────────┬───────────┘
                               │
                               ▼
         ┌─────────────────────────────────────────┐
         │           ② 智能匹配算法                 │
         │           OrderMatchAlgorithm            │
         │                                         │
         │   距离(40%) + 时间(30%) + 信誉(30%)      │
         │        ↓                   ↑             │
         │   匹配分排序           信誉分输入          │
         └────────┬────────────────┬───────────────┘
                  │                │
                  ▼                │
         ┌──────────────┐  ┌──────┴───────┐
         │ 跑腿员接单    │  │ ③ 信誉积分    │
         │ 并完成配送    │  │   系统        │
         └──────┬───────┘  │              │
                │          │ +2 按时完成   │
                └─────────►│ -3 超时完成   │
                           │ -5 违规取消   │
                           └──────────────┘
```

**完整正向循环**：

```
合理定价(智能定价) → 订单吸引力高 → 跑腿员积极接单
       ↑                                    │
       │                                    ▼
  更多优质订单 ← 推荐排名靠前 ← 信誉加分 ← 按时完成
```

| 联动关系 | 说明 |
|----------|------|
| **定价 → 匹配** | 合理的建议价格确保订单不会因价格过低而无人问津，提升匹配效率 |
| **匹配 → 信誉** | 匹配算法将信誉分作为 30% 权重因子，高信誉跑腿员获得更靠前的推荐 |
| **信誉 → 匹配** | 信誉变更直接影响下一次匹配的得分高低，形成实时反馈 |
| **定价中的距离 → 匹配中的距离** | 两者都依赖地理坐标：定价用步行路径距离，匹配用 Haversine 直线距离 |

**正向循环**：合理定价 → 订单有吸引力 → 按时完成 → 信誉加分 → 推荐排名靠前 → 接到更多好订单

**反向惩罚**：超时/违规 → 信誉扣分 → 推荐排名靠后 → 可用优质订单减少

这种三位一体的机制确保了平台生态的健康运转，激励跑腿员提供高质量服务。
